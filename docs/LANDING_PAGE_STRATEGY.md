# FOM Play Landing Page Strategy

Last Updated: 2026-04-19 (Asia/Jakarta)
Owner: Product/Engineering (FOM Play)

## 1. Goal
Build an app-first hybrid homepage on `https://fomplay.asia/` that:

- helps new users understand what FOM Play is
- keeps fast access to the app for returning users
- creates indexable SEO entry points for padel-related search intent
- highlights FOM Play's real product differentiators

## 2. URL Strategy

Primary recommendation:

- `/` = hybrid homepage
- `/app` = main app shell
- `/fitur` = feature overview
- `/format/americano` = SEO page
- `/format/mexicano` = SEO page
- `/format/match-play` = SEO page
- `/ranking` = ranking explanation page
- `/faq` = FAQ page
- `/edukasi/perbedaan-americano-vs-mexicano` = comparison article

Notes:

- Root remains the most memorable URL for users.
- Root should still push users quickly into the app.
- Public pages should be indexable and linked from `/`.
- Existing shared-link behavior like `?shared=...` should keep working during migration.

## 3. Positioning

Recommended category statement:

> FOM Play is a padel app for hosting games, live scoring, shareable scoreboards, and player rankings.

Differentiators to emphasize:

- supports Americano, Mexicano, and Match Play
- players can view results from their own phones
- live scoring and automatic standings
- scoreboard visuals suitable for Instagram Story sharing
- global and regional rankings
- lower friction than products that require every player to log in first

Avoid over-emphasizing:

- "scores are lost in WhatsApp"

Reason:

- this is not the strongest pain point from current product understanding
- better to focus on host workflow, sharing, aesthetics, and ranking

## 6. React Component Plan

Current app observation:

- app uses a large single `App.tsx`
- navigation is screen-state based, not router-based
- current app flow is tightly coupled to internal screen state

Recommended implementation approach:

- keep internal app screen logic for `/app`
- add a thin path-based shell above the current app flow
- do not refactor the whole app into router-first navigation in one step

Suggested component split:

- `src/App.tsx`
  - route resolver
  - chooses public page vs app shell
- `src/pages/HomePage.tsx`
- `src/pages/FeaturesPage.tsx`

## 8. UX Rules for Hybrid Root

- if user is authenticated, use copy like `Lanjutkan ke App`

## 9. SEO Rules

Minimum SEO requirements for `/`:

- one clear H1
- real body copy, not only UI labels
- internal links to public routes
- unique `title`
- unique `meta description`
- canonical URL

Recommended schema:

- `SoftwareApplication`
- `FAQPage` on `/faq` and possibly root if FAQ is present
- `Organization`

Recommended internal linking:

- homepage links to `/fitur`, `/format/americano`, `/format/mexicano`, `/ranking`, `/faq`
- feature and format pages link back to `/app`
- FAQ answers link to related format pages where relevant

## 10. MVP Build Order

Recommended order:

1. create route-aware shell with `/` and `/app`
2. build hybrid homepage
3. build `/fitur`
4. build `/format/americano`
5. build `/format/mexicano`
6. build `/ranking`
7. build `/faq`

Why this order:

- gives product clarity quickly
- creates SEO entry points early
- avoids blocking on full content expansion

## 11. Implementation Notes for Existing Codebase

Current codebase details that matter:

- page title and canonical behavior currently assume app on root
- analytics page tracking uses app screen names
- app currently starts from `login` or shared view based on query string

Migration implications:

- update root metadata to match hybrid homepage
- keep app analytics for `/app`
- add page-view tracking for public pages
- preserve current shared-view links and active app flows

## 12. Decision Summary

Chosen direction:

- use `fomplay.asia/` as a hybrid homepage
- keep FOM Play app accessible from a strong CTA and move main app shell to `/app`
- build a small set of public pages for SEO and education
- emphasize host workflow, sharing, aesthetics, and ranking as core differentiation
